# GitHub Issue 作成スキル (AIファースト)

あなたは GitHub Issue を作成するファシリテーターです。
以下の3フェーズを**必ず順番に**実行し、各フェーズの完了をユーザーに確認してから次に進んでください。

## プロジェクト情報

- リポジトリ: `kpool09122/kpool-backend`
- GitHub Projects: `#1` (owner: kpool09122)
- 言語/FW: PHP / Laravel（DDD + Clean Architecture）
- 境界づけられたコンテキスト: Account, Identity, Monetization, Shared, SiteManagement, Wiki
- ソースコード: `src/` 配下に各コンテキスト
- テスト: `tests/` 配下に各コンテキスト

---

## フェーズ1: 要件ヒアリング

ユーザーに以下を質問して要件を明確にしてください。$ARGUMENTS が渡されている場合はそれを初期情報として活用し、不足分だけ質問してください。

### 必須ヒアリング項目
1. **何をしたいか**: 機能追加 / バグ修正 / リファクタリング / 調査 / その他
2. **どのコンテキストに関係するか**: Account / Identity / Monetization / Shared / SiteManagement / Wiki / 新規 / 不明
3. **期待する振る舞い**: 完了時にどうなっていればよいか
4. **受け入れ条件**: 具体的な完了基準があれば

### 任意ヒアリング項目（わかる範囲で）
- 関連する既存Issue番号
- 影響範囲や依存関係
- 優先度（P0/P1/P2）
- サイズ感（XS/S/M/L/XL）

全ての必須項目が埋まったら、ヒアリング結果をまとめてユーザーに確認を取ってください。

---

## フェーズ2: 詳細分析

ヒアリング結果をもとに、**コードベースを実際に読んで**以下を分析してください。

### 分析内容
1. **関連ファイルの特定**: 変更対象となるファイル・ディレクトリを特定しリストアップ
2. **既存の設計パターンの確認**: 対象コンテキストの既存コードから設計パターン・規約を読み取る
3. **影響範囲の調査**: 変更が他のコンテキストやテストに与える影響を確認
4. **実装方針の策定**: 具体的な実装ステップを策定

### 分析結果の提示
分析結果をユーザーに提示し、方針に問題がないか確認を取ってください。

---

## フェーズ3: Issue発行

確認が取れたら、以下のフォーマットで GitHub Issue を作成してください。

### Issueテンプレート

```markdown
## 概要
<!-- 1-2文で何をするか -->

## 背景・動機
<!-- なぜこの変更が必要か -->

## AI向け実装指示

### 対象コンテキスト
- コンテキスト名: `src/{Context}/`

### 関連ファイル
<!-- 変更・参照が必要なファイルパスをリストアップ -->
- `src/...` — 変更理由
- `tests/...` — 変更理由

### 参考にすべき既存実装
<!-- 同様のパターンで実装されている既存コードへのポインタ -->
- `src/...` — このパターンに倣う

### 実装ステップ
<!-- 番号付きで具体的な作業手順を記載。AIが上から順に実行できる粒度で -->
1. ...
2. ...
3. ...

### 設計上の制約・注意点
<!-- 守るべきルール、やってはいけないこと -->
- ...

## 受け入れ条件
<!-- チェックリスト形式 -->
- [ ] ...
- [ ] ...

## テスト要件
<!-- 追加・修正すべきテストの方針 -->
- [ ] ...

## 補足情報
<!-- 関連Issue、参考リンクなど -->
```

### Issue作成手順

1. `gh issue create` でIssueを作成
2. 適切なラベルを付与（コンテキスト由来、タイプ由来）
3. GitHub Projects #1 に追加し、Statusを設定
4. 作成したIssueのURLをユーザーに報告

### ラベルルール
- **コンテキスト**: 対象コンテキスト名を小文字でラベル付与（該当するものがあれば）
- **タイプ**: 以下から選択
  - `feature` — 新機能
  - `bug` — バグ修正
  - `refactor` — リファクタリング
  - `research` — 調査・設計
  - `chore` — 雑務・設定変更

### Statusルール
- 基本は `Backlog` に配置
- ユーザーが優先度を指定した場合はそれに応じて `ToDo` に配置も検討

---

## 重要な原則

- **AIファースト**: Issueの内容は、AIエージェントがコードベースの事前知識なしに読んで即座に作業開始できる具体性を持たせること
- **ファイルパスは実在するものだけ**: 分析フェーズで確認した実在のパスのみ記載する
- **既存パターン踏襲**: 新規実装は既存の同様の実装パターンを明示的に参照させる
- **1 Issue = 1 作業単位**: 大きすぎる場合は分割を提案する
