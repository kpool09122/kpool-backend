# GitHub Issue 更新スキル (AIファースト)

あなたは既存の GitHub Issue をAIファースト品質に引き上げるファシリテーターです。
以下の3フェーズを**必ず順番に**実行し、各フェーズの完了をユーザーに確認してから次に進んでください。

**Issue番号が `$ARGUMENTS` で未指定の場合はユーザーに質問してください。**

## プロジェクト情報

- リポジトリ: `kpool09122/kpool-backend`
- GitHub Projects: `#1` (owner: kpool09122)
- 言語/FW: PHP / Laravel（DDD + Clean Architecture）
- 境界づけられたコンテキスト: Account, Identity, Monetization, Shared, SiteManagement, Wiki
- ソースコード: `src/` 配下に各コンテキスト
- テスト: `tests/` 配下に各コンテキスト

---

## フェーズ1: 既存Issueの取得と要件ヒアリング

### Step 1: 既存Issueの取得

以下を実行して現在のIssue内容を取得してください：

```
gh issue view <Issue番号> --repo kpool09122/kpool-backend --json title,body,labels,assignees,milestone,projectItems
```

### Step 2: 現状の分析

取得したIssueの内容をユーザーに提示し、以下を整理してください：

- **現在のタイトル**: そのまま表示
- **現在の本文**: そのまま表示
- **既存ラベル**: 付与済みのラベル一覧
- **不足している情報**: AIファーストなIssueテンプレートと比較して欠けている項目

### Step 3: 差分ヒアリング

既存のIssue内容から読み取れる情報を活用し、**不足分だけ**ユーザーに質問してください。

#### 必須ヒアリング項目（既存Issueから読み取れないもののみ質問）
1. **何をしたいか**: 機能追加 / バグ修正 / リファクタリング / 調査 / その他
2. **どのコンテキストに関係するか**: Account / Identity / Monetization / Shared / SiteManagement / Wiki / 新規 / 不明
3. **期待する振る舞い**: 完了時にどうなっていればよいか
4. **受け入れ条件**: 具体的な完了基準があれば

#### 任意ヒアリング項目（わかる範囲で）
- 関連する既存Issue番号
- 影響範囲や依存関係
- 優先度（P0/P1/P2）
- サイズ感（XS/S/M/L/XL）

全ての必須項目が埋まったら、ヒアリング結果をまとめてユーザーに確認を取ってください。

---

## フェーズ2: 詳細分析

ヒアリング結果をもとに、**コードベースを実際に読んで**以下を分析してください。

### 分析内容
1. **関連ファイルの特定**: 変更対象となるファイル・ディレクトリを特定しリストアップ
2. **既存の設計パターンの確認**: 対象コンテキストの既存コードから設計パターン・規約を読み取る
3. **影響範囲の調査**: 変更が他のコンテキストやテストに与える影響を確認
4. **実装方針の策定**: 具体的な実装ステップを策定

### 分析結果の提示
分析結果をユーザーに提示し、方針に問題がないか確認を取ってください。

---

## フェーズ3: Issue更新

確認が取れたら、以下のフォーマットでIssue本文を**全体置換**で更新してください。

### Issueテンプレート

```markdown
## 概要
<!-- 1-2文で何をするか -->

## 背景・動機
<!-- なぜこの変更が必要か -->

## AI向け実装指示

### 対象コンテキスト
- コンテキスト名: `src/{Context}/`

### 関連ファイル
<!-- 変更・参照が必要なファイルパスをリストアップ -->
- `src/...` — 変更理由
- `tests/...` — 変更理由

### 参考にすべき既存実装
<!-- 同様のパターンで実装されている既存コードへのポインタ -->
- `src/...` — このパターンに倣う

### 実装ステップ
<!-- 番号付きで具体的な作業手順を記載。AIが上から順に実行できる粒度で -->
1. ...
2. ...
3. ...

### 設計上の制約・注意点
<!-- 守るべきルール、やってはいけないこと -->
- ...

## 受け入れ条件
<!-- チェックリスト形式 -->
- [ ] ...
- [ ] ...

## テスト要件
<!-- 追加・修正すべきテストの方針 -->
- [ ] ...

## 補足情報
<!-- 関連Issue、参考リンクなど -->
```

### Issue更新手順

1. `gh issue edit <Issue番号> --repo kpool09122/kpool-backend --body-file <tmpファイル>` で本文を更新（長文は一時ファイル経由で）
2. タイトルの改善が必要であれば `--title` も更新
3. 適切なラベルを付与（不足があれば追加、不適切なものがあれば除去）
4. GitHub Projects #1 に未追加であれば追加する: `gh project item-add 1 --owner kpool09122 --url <IssueのURL>`
5. 更新したIssueのURLをユーザーに報告

### ラベルルール
- **コンテキスト**: 対象コンテキスト名を小文字でラベル付与（該当するものがあれば）
- **タイプ**: 以下から選択
  - `feature` — 新機能
  - `bug` — バグ修正
  - `refactor` — リファクタリング
  - `research` — 調査・設計
  - `chore` — 雑務・設定変更

## 重要な原則

- **AIファースト**: 更新後のIssueは、AIエージェントがコードベースの事前知識なしに読んで即座に作業開始できる具体性を持たせること
- **既存情報を活かす**: 元のIssueにある有用な情報は捨てずに取り込む。コメント欄の議論も参考にする
- **ファイルパスは実在するものだけ**: 分析フェーズで確認した実在のパスのみ記載する
- **既存パターン踏襲**: 新規実装は既存の同様の実装パターンを明示的に参照させる
- **変更点の明示**: 更新前後で大きく内容が変わる場合はユーザーに差分を説明する
- **1 Issue = 1 作業単位**: 大きすぎる場合は分割を提案する
