# GitHub PR 作成スキル

あなたは GitHub Pull Request を作成するアシスタントです。
`.github/pull_request_template.md` のテンプレートをベースに、バックエンドリポジトリに最適化したPRを作成します。

## 設定

- リポジトリ: `kpool09122/kpool-backend`
- ベースブランチ: ユーザーが指定したブランチを優先し、未指定なら `main`
- テンプレート: `.github/pull_request_template.md`

## バックエンド固有の除外項目

以下のセクションはバックエンドリポジトリでは不要なため省略します。

- 📸 スクリーンショット・デモ
- 💄 UI/UX (Style) の変更種類チェック
- 動作確認（手動テスト記載欄）

## 実行手順

### Step 1: 現在の状態を把握

以下を並列で確認します。

- `git status` で未コミットの変更を確認
- `git log` で現在のブランチのコミット履歴を確認
- `git diff <ベースブランチ>...HEAD` でベースブランチからの全差分を確認
- 現在のブランチがリモートにpush済みか確認

未コミットの変更がある場合は報告し、PR作成を続ける前に扱いを確認します。

### Step 2: 差分を分析

コミット履歴と差分を読み、以下を整理します。

- 変更内容の要約
- 変更の種類: 新機能 / バグ修正 / リファクタリング / ドキュメント / テスト / ビルドCI / パフォーマンス / 削除
- 変更理由・背景
- レビューポイント
- 注意事項: 破壊的変更、マイグレーション等

### Step 3: 関連Issueの確認

関連Issue番号は必ずユーザーに確認します。最低限、以下を聞きます。

- `Closes #...`
- `Fixes #...`
- `Relates to #...`

該当なしなら「なし」として扱います。

### Step 4: PR本文を構成して提示

テンプレートに沿ってPR本文を構成し、タイトル案と本文案をユーザーに提示して確認を取ります。
タイトルは70文字以内で簡潔にまとめます。

### Step 5: PR作成

確認後、以下を実行します。

1. リモートにpushされていなければ `git push -u origin <branch>` で push
2. `gh pr create` でPRを作成
3. 作成したPRのURLを報告

```markdown
- [ ] 📚 ドキュメント (Documentation)
- [ ] 🧪 テスト (Tests)
- [ ] 🔨 ビルド/CI (Build/CI)
- [ ] ⚡ パフォーマンス (Performance)

- [ ] `make check` を実行し、すべてのテストがパスすることを確認
- [ ] 新しく追加した機能に対するテストを作成
- [ ] 既存のテストが壊れていないことを確認

## 🔍 レビューのポイント

<!-- 差分分析から自動生成 -->

## 📖 関連情報

### 関連Issue・タスク

Closes #
Fixes #
Relates to #

## ⚠️ 注意事項

<!-- 破壊的変更、マイグレーション等があれば記載。なければ「特になし」 -->

---

## チェックリスト

- [ ] 自分でコードレビューを実施した
- [ ] 適切なブランチ名を使用している
- [ ] コミットメッセージが適切である
- [ ] 必要に応じてドキュメントを更新した
- [ ] 破壊的変更がある場合は適切に文書化した
```

## 重要な原則

- 関連Issueは必ず確認する
- コミットメッセージだけでなくコード差分を実際に読む
- 該当する変更種類だけ `[x]` にする
- 関連Issueが「なし」の場合は不要な行を削除する
- 不要セクションは出力しない
