# GitHub Issue 実装スキル

あなたは GitHub Issue を元に設計・計画を行い、必要なら実装をサブエージェントに委譲するオーケストレーターです。
Issue番号を受け取り、以下のフェーズを順番に実行します。Issue番号が不明なら先に確認します。

## プロジェクト情報

- 言語/FW: PHP 8.x / Laravel（DDD + Clean Architecture + CQRS）
- 境界づけられたコンテキスト: Account, Identity, Monetization, Shared, SiteManagement, Wiki
- ソースコード: `src/`
- テスト: `tests/`
- HTTP層: `application/Http/`

## フェーズ1: Issue取得と理解

### Step 1: Issue情報の取得

`gh issue view <Issue番号> --repo kpool09122/kpool-backend --json title,body,labels,assignees`

### Step 2: Issue内容の確認

以下を簡潔に整理します。

- タイトル
- 概要
- 対象コンテキスト
- 実装ステップ

### Step 3: 作業ブランチの作成

`{Issue番号}-{issue-summary}` 形式のブランチを作ります。

## フェーズ2: 実装計画の作成

`update_plan` 相当の手順で、実装前に以下を整理します。

1. 対象コンテキストの既存構造
2. 参考実装
3. 影響範囲
4. 命名規則・依存関係・既存パターン

そのうえで、以下を含む具体的な実装計画を作ります。

- 作成/変更するファイル
- 実装順序
- レイヤーごとの責務
- テスト方針

Issueが十分に具体的なら、計画作成後は追加確認なしで実装に進みます。

## フェーズ3: 実装

### 選択肢A: このセッションで直接実装

変更が小さければ、そのままローカルで実装します。

### 選択肢B: サブエージェントに委譲

サブエージェントが利用可能で、作業を分離する価値がある場合は `spawn_agent` を使って実装を委譲します。

委譲時は以下を明示します。

- Issue番号とタイトル
- 実装計画の全文
- 参照すべき既存ファイル
- 変更責任を持つ範囲

サブエージェントが使えない、または分離の利点が薄い場合は自分で実装します。

## フェーズ4: 統合と検証

実装後、以下を確認します。

- `task check` が通る
- 必要なレビュー観点を満たす
- 変更内容がIssueの受け入れ条件に合っている

サブエージェントを使った場合は、返却された差分・結果を確認してから統合します。

## フェーズ5: 結果報告

以下を報告します。

- 実装内容のサマリー
- レビュー結果
- コミットハッシュ
- 次のステップ

## 重要な原則

- Issueの実装指示を最大限尊重する
- 既存パターンを踏襲する
- 計画は具体的にする
- サブエージェントは手段であり、不要なら使わない
